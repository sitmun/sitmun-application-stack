# SITMUN Seed Data

Source-of-truth data files for seed data and translations.

**Key principle**: Development profile CSVs are the source of truth. Production profiles (postgres, oracle) receive generated, version-controlled outputs. Never edit production CSVs or YAML changesets manually.

Generated CSV and YAML use LF line endings only (scripts pass `newline="\n"` / `newline=""` so output is LF on all platforms; otherwise Windows would write CRLF).

## Files

| File | Purpose |
|------|---------|
| `master-seed-data.json` | Canonical non-translation seed data (English, generated by `extract_seed_data.py`) |
| `master-i18n.<lang>.json` | Immutable per-language translation baselines |
| `i18n-active-baseline.json` | Active baseline selector (`{ "activeBaseline": "en" }`) |

Python tooling lives in [`tools/bin/`](../bin/). Shell scripts live in [`tools/scripts/`](../scripts/). Integration tests live in [`tools/tests/`](../tests/).

## I18n model

Translation baselines are per-language, immutable JSON files:

```
tools/seed-data/
  master-i18n.en.json         # English baseline (canonical default)
  master-i18n.es.json         # Spanish baseline
  master-i18n.ca.json         # Catalan baseline
  master-i18n.fr.json         # French baseline
  master-i18n.oc-aranes.json  # Aranese baseline
  i18n-active-baseline.json   # Selector: { "activeBaseline": "en" }
```

**Editing translations**: edit the corresponding `master-i18n.<lang>.json` directly.

**Adding a new translatable entry**: add rows to all baselines, then re-run `generate_all_seed_outputs.py`.

**Baseline resolution precedence** (all scripts):
1. `--baseline <lang>` CLI argument
2. `i18n-active-baseline.json` `activeBaseline` field
3. Hard error — no implicit fallback

## Seed data model

`master-seed-data.json` is English/default canonical state. It is never modified by language switching. It contains all non-translation entities: `STM_CODELIST`, `STM_TSK_TYP`, `STM_TER_TYP`, `STM_USER`, `STM_TASK`, `STM_TSK_UI`, `STM_GRP_TSK`, `STM_CONF`.

## Entity classification

| Classification | Entities | Sync policy |
|---|---|---|
| **Extensible** | All entities | Upsert only (user-created rows preserved) |

All entities use upsert-only policy. No seed data operation deletes rows, because user-created entries (custom codelist values, task types, config parameters) must survive upgrades.

## End-to-end workflow

### Normal cycle (seed data or translation change)

```bash
# 1. Edit development seed CSVs in profiles/development/backend/liquibase/changelog/
#    (source of truth — never touch postgres/oracle CSVs directly)

# 2. Re-extract seed data master
python3 tools/bin/extract_seed_data.py

# 3. If translatable columns changed: also re-extract translations
python3 tools/bin/extract_translatable.py --scenario development --baseline en

# 4. Edit master-i18n.*.json for any new/changed translatable entries
#    (add the new entries to all baselines, in all languages)

# 5. Regenerate production profiles
python3 tools/bin/generate_all_seed_outputs.py --baseline en
```

### Language switching (production default language)

```bash
# Preview
python3 tools/bin/switch_default_language.py es --dry-run

# Switch production profiles to Spanish default
python3 tools/bin/switch_default_language.py es

# Switch back to English
python3 tools/bin/switch_default_language.py en
```

No files are swapped; the baseline is selected and generators run. Development is never touched.

### Applying the changelog to a remote database

```bash
bash tools/scripts/apply-seed-data.sh \
  --db-type postgres \
  --url jdbc:postgresql://db.example.com:5432/sitmun \
  --username sitmun3 --password secret

bash tools/scripts/apply-seed-data.sh \
  --db-type oracle \
  --url "jdbc:oracle:thin:@//oracle.example.com:1521/sitmun" \
  --username sitmun3 --password secret \
  --baseline es
```

### Adding a new production scenario

```bash
mkdir -p profiles/newscenario/liquibase/changelog/{02_codelists,03_task_types,04_seed_data,05_translations,06_params}

# Scenario is auto-discovered from filesystem — no code changes needed
python3 tools/bin/generate_all_seed_outputs.py --baseline en --scenarios newscenario
```

## Commands reference

```bash
# Generate all production outputs (default: postgres,oracle)
python3 tools/bin/generate_all_seed_outputs.py --baseline en

# Generate for a single scenario
python3 tools/bin/generate_all_seed_outputs.py --baseline en --scenarios oracle

# Generate only translations
python3 tools/bin/generate_all_seed_outputs.py --baseline en --only translations

# Generate only seed data
python3 tools/bin/generate_all_seed_outputs.py --only seed

# Check translation completeness
python3 tools/bin/check_translations.py --baseline en

# Export missing translations to CSV for bulk completion
python3 tools/bin/check_translations.py --baseline en
#   Edit missing-translations.csv, then:
python3 tools/bin/import_missing_translations.py --baseline en

# Re-extract seed data from development CSVs
python3 tools/bin/extract_seed_data.py

# Re-extract translatable content
python3 tools/bin/extract_translatable.py --scenario development --baseline en

# Switch default language
python3 tools/bin/switch_default_language.py es

# Generate for a specific scenario directly
python3 tools/bin/generate_translation_files.py --scenario oracle --baseline ca
python3 tools/bin/generate_seed_files.py --scenario oracle
```

## Liquibase strategy

Generated YAML changesets use `loadUpdateData` with `runOnChange: true`.

- **Reference entities**: upsert + post-load cleanup (DELETE rows not in seed).
- **Extensible entities**: upsert only (no deletion — user-created rows survive upgrades).
- **Sequence sync**: every changeset ends with `UPDATE STM_SEQUENCE SET SEQ_COUNT = (SELECT COALESCE(MAX(PK), 0) + 1 FROM table) WHERE SEQ_NAME = 'PK'`.
- **FK safety**: `STM_TRANSLATION` cleanup runs before `STM_LANGUAGE` cleanup.

## Safety rules

- **Never manually edit** postgres/oracle seed CSVs or YAML changesets — they are generated files.
- **Never create** hand-written migration YAMLs for seed data in production profiles.
- **Never run** `generate_seed_files.py --scenario development` (rejected by the tool).
- **Development profile** is never modified by any tool in this directory.
- `master-seed-data.json` always remains English/default canonical state.
- Edit translations by modifying the appropriate `master-i18n.<lang>.json` file directly.
